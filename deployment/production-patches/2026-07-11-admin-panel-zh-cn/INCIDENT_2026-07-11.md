# Admin Panel Build Incident And Recovery Gate

Date: 2026-07-11

Status: host recovered; fourth image build stopped safely; Chinese Admin Panel not deployed.

## What Happened

An isolated dependency image was used on the production host to run the pinned
Admin Panel source checks. A complete ESLint pass ran without explicit CPU,
memory, or wall-clock limits. While it was running, the host stopped completing
SSH banner exchange and public HTTPS TLS handshakes.

The temporary check container used the image ancestor:

```text
librechat-admin-panel-zh-cn-deps:1f409f3
```

An SSH cleanup attempt could not complete after the host became unresponsive,
so the container's final state is not confirmed.

## Confirmed Non-Changes

- `scripts/deploy.sh` was not run.
- `/opt/librechat/compose.override.yaml` was not intentionally changed.
- No production service was intentionally recreated or replaced.
- The Chinese Admin Panel image was not completed or deployed.
- LibreChat, Office, CodeAPI, MongoDB data, model configuration, uploads, and
  conversations were not intentionally modified by this build attempt.

Temporary files, Docker build cache, and an incomplete temporary image may
remain on the host.

## Repository Corrections

- `74cb0cb` fixes source formatting and import-order gates against the pinned
  upstream baselines.
- `1a438fe` makes production image builds use a disposable BuildKit container
  with default limits of 1.25 GiB, 0.75 CPU, and 45 minutes.
- Both commits were pushed to `origin/main` before any further remote work.

## Required Recovery Gate

After the host is reachable, do not build first. Stop any surviving temporary
check container and then perform read-only inspection:

```bash
ids=$(docker ps -q --filter ancestor=librechat-admin-panel-zh-cn-deps:1f409f3)
test -z "$ids" || docker kill $ids
uptime
free -h
df -h
docker ps
journalctl -k --since '-60 min' | tail -200
```

Before a new build, confirm all of the following:

- the main site and `/api/config` respond normally;
- the Admin Panel still runs the official pre-change image digest;
- `/office/` returns `401` with realm `Office Converter`;
- CodeAPI is healthy;
- MongoDB `configs` count remains `0`;
- API, Nginx, CodeAPI, and MongoDB were not unexpectedly recreated.

Only after those checks pass may the committed resource-limited build run.
Deployment must still recreate only `LibreChat-Admin-Panel` and must record the
immutable image ID, backup directory, protected container IDs, and acceptance
results.

## Resource-Limited Build Attempt R2

The recovery gate passed and a second build ran with a disposable BuildKit
container limited to 1 GiB memory, no additional swap, 0.5 CPU, and 45 minutes.
The limits were verified from the running container. The main site and Admin
Panel remained HTTP 200, while Office remained the expected authenticated 401.

The build failed closed before lint, tests, image output, or deployment because
`.dockerignore` excluded `scripts/sort-imports.ts`, which
`scripts/check-import-baseline.mjs` requires inside the Docker build context.
No candidate image, `BUILD_RESULT.txt`, or `DEPLOY_RESULT.txt` was produced, and
the official Admin Panel and all protected production containers were unchanged.

The repository correction explicitly includes `scripts/sort-imports.ts` in the
Docker build context and makes `verify-source.sh` reject any future release that
omits either that file or its `.dockerignore` exception. A new immutable source
hash and image tag are required before another production build.

## Resource-Limited Build Attempt R3

Release commit `80f0bbb` corrected the Docker context omission and passed the
remote source gate. The third build used the same verified 1 GiB memory, no
additional swap, 0.5 CPU, and 45-minute limits. The import-order baseline passed
inside Docker, confirming the R2 correction.

The single full-tree ESLint process then reached the 1 GiB BuildKit cgroup limit
and was terminated after about 10 minutes with `SIGKILL`, `ResourceExhausted`,
and `cannot allocate memory`. The disposable builder was removed automatically.
No candidate image, `BUILD_RESULT.txt`, or deployment was produced. Throughout
the attempt the main site and Admin Panel remained HTTP 200 and Office remained
the expected authenticated 401.

The next repository correction keeps strict ESLint coverage and
`--max-warnings 0`, but enumerates every non-generated TypeScript source file
and checks deterministic sequential batches in separate Node processes. Each
process has a 512 MiB old-space limit, so completed batches release memory while
the outer BuildKit container remains capped at 1 GiB. The release preflight also
verifies the lint plan and that its runner is present in the Docker context.

## Resource-Limited Build Attempt R4

Release commit `2058a6e` changed strict ESLint to deterministic batches, but the
runner executed under Bun because the pinned Bun image maps `node` to Bun. The
first 24-file batch again approached the 1 GiB BuildKit limit. The transient
build was stopped deliberately before OOM; its builder was removed, no image or
result file was produced, and production health remained unchanged.

Bounded probes then established the actual runtime behavior:

- Bun exposes `--smol`, but both 24-file and 4-file batches still hit the 1 GiB
  container limit while evaluating the import dependency graph.
- Alpine provides pinnable `nodejs=22.23.0-r0`; real Node honored V8 heap limits,
  but 4 files at 384 MiB and 24 files at 512 MiB still exhausted their heaps.
- Raising the container ceiling to 1.25 GiB did not solve the V8 heap limit and
  is therefore not a valid production-host strategy.

The durable correction moves complete lint, typecheck, unit tests, and build to
GitHub Actions. The pinned upstream revision already has a successful official
GitHub Actions build. The modified source must also pass its repository workflow
and receive an immutable `admin-ci-<source-hash>` tag before production packaging.
The production Docker build repeats only deterministic lightweight gates and
builds the exact source hash covered by that CI attestation.

## GitHub Actions Verification Attempt R5

Commit `984c0a9` started repository workflow run `29147181959`. Source
verification, dependency installation, locale checks, formatting, import order,
strict ESLint, and typecheck completed. The unit-test stage reported 757 passing
tests and two localization-test failures, so the workflow stopped before the
application build and did not publish an `admin-ci-<source-hash>` tag.

Both failures initially appeared to be test-isolation defects rather than a
production deployment:

- the first `LanguageSelector` test assumed the mutable shared i18n singleton
  was already at its default language before rendering;
- the Help source-link test coupled link discovery to the Chinese accessible
  label even though the link is intentionally available in both languages.

The correction resets the selector test to Simplified Chinese before every test
and verifies the Help source link under either supported localized label. No
production command, image build, Compose change, container recreation, or
cleanup ran as part of R5. R8 records the later runtime root-cause finding that
superseded the initial test-isolation diagnosis.

## GitHub Actions Verification Attempt R6

Commit `10c5487` started repository workflow run `29147815760`. The pinned
source and locale checks passed and the locked dependencies installed. The
quality gate then stopped at the format baseline because the modified Help test
needed one deterministic Prettier line break. No lint, typecheck, unit test,
application build, tag publication, production build, or deployment ran.

The same run showed that the current GitHub runner does not include `rg`.
Because the credential-pattern command was inside a shell conditional, the
missing executable printed an error but did not fail the source gate. The
correction replaces that runner dependency with a repository-owned Python
scanner that fails closed and runs identically on CI and the production host.

## GitHub Actions Verification Attempt R7

Commit `a9ad769` started repository workflow run `29148070450`. The source,
locale, credential, and Prettier gates passed. The import baseline then stopped
on the modified `LanguageSelector` test because its two external imports were
not ordered shortest-first as required by the pinned upstream sorter. The

## GitHub Actions Verification Attempt R8

Commit `65f540f` started repository workflow run `29148387606`. Source, locale,
credential, formatting, import order, strict ESLint, and typecheck passed. The
unit-test stage reported 758 passing tests and one failure: even after an
explicit change to `zh-Hans`, the shared i18n instance resolved to English.
The application build and tag publication did not run, and production remained
unchanged.

Inspection of the pinned `i18next@25.10.10` implementation established the
runtime root cause. With `nonExplicitSupportedLngs: true`, `isSupportedCode`
reduces `zh-Hans` to `zh` before checking `supportedLngs`. This release registers
the exact resources `zh-Hans` and `en`, not `zh`, so the intended default was
rejected and deterministically fell back to English. The correction disables
that reduction and adds regression assertions for both the resolved language
code and the visible Simplified Chinese label.

## GitHub Actions Verification Attempt R9

Commit `19d5d0e` started repository workflow run `29148790345`. The source and
locale gates passed, but the format baseline stopped on `locales.test.ts` before
import order, lint, typecheck, unit tests, build, or tag publication. The file
had been wrapped to an 80-column assumption, while the pinned project
configuration uses `printWidth: 100`; the 86-character import therefore belongs
on one line. The correction follows the repository configuration exactly. No
production command or cleanup ran in R9.
